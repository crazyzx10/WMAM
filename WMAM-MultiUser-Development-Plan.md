# WMAM 多用户版本开发方案

**日期**：2026-05-24  
**状态**：已确认基础方案、UI 方向和技术选型，待后续开发拆分  
**适用范围**：基于现有 `go-app` 的多用户在线版本改造

---

## 1. 项目目标

将当前本地单人使用的 WMAM 工具，改造成运行在云服务器上的多人在线 Web 应用。

第一版只聚焦以下目标：

1. 多人在线使用。
2. 管理员统一维护系统配置、小程序配置和用户。
3. 普通用户只能执行拉取相关操作。
4. 拉取广告数据写入外部 MySQL 数据库。
5. WMAM 自身的用户、权限、配置、任务摘要和审计日志存储在应用服务器本地。

第一版不做：

1. 定时自动拉取。
2. 广告数据看板或报表。
3. 多套 MySQL 数据库配置。
4. 配置批量导入导出给普通用户使用。
5. 多管理员账号。

---

## 2. 部署架构

目标架构：

```text
管理员电脑 / 普通用户电脑
  ↓ 浏览器访问
云服务器 A：运行 WMAM Web 应用
  ├─ 本地系统存储：用户、权限、小程序配置、数据库连接配置、任务摘要、操作审计日志
  └─ 执行拉取时连接
MySQL 服务器 B：保存微信广告拉取结果数据
```

关键约定：

1. WMAM 运行在云服务器 A。
2. 广告数据 MySQL 可以在另一台服务器 B。
3. WMAM 启动后不主动连接 MySQL。
4. 只有在管理员测试数据库连接，或用户执行拉取时，才连接 MySQL。
5. MySQL 服务器 B 只作为广告数据落库目标，不保存 WMAM 自身的账号、权限和系统配置。

---

## 3. 本地系统存储

WMAM 云服务器 A 上需要有一个本地系统存储，用于保证即使广告数据 MySQL 暂时不可用，管理员仍然可以登录系统、修改数据库配置、恢复配置。

建议第一版使用 SQLite 文件作为本地系统存储。

本地系统存储保存：

1. 用户与权限。
2. 小程序配置。
3. 数据库连接配置。
4. 上一份可用数据库连接配置。
5. 任务摘要。
6. 操作审计日志。
7. 管理员恢复码哈希。
8. 系统初始化状态。

本地系统存储不保存：

1. 微信广告明细数据。
2. 执行过程中的详细实时日志。
3. 明文用户密码。
4. 30 天“记住密码”的登录状态备份。

---

## 4. 角色与权限

第一版只有两种角色：

1. 管理员 `admin`
2. 普通用户 `user`

系统只允许一个管理员账号。

### 4.1 管理员权限

管理员可以：

1. 登录系统。
2. 修改系统配置。
3. 修改数据库连接配置。
4. 测试数据库连接。
5. 恢复上一份可用数据库配置。
6. 导出系统配置。
7. 导入系统配置。
8. 添加、编辑、删除、启用、禁用小程序。
9. 创建、启用、禁用普通用户。
10. 重置普通用户密码。
11. 执行拉取。
12. 中断、继续、结束任何人的任务。
13. 查看所有历史任务摘要。
14. 查看所有操作审计日志。

管理员不能：

1. 创建第二个管理员。
2. 删除自己。
3. 禁用自己。
4. 将自己降级为普通用户。

### 4.2 普通用户权限

普通用户只能：

1. 登录系统。
2. 查看执行拉取页面。
3. 执行拉取。
4. 中断自己发起的任务。
5. 继续自己发起的任务。
6. 结束自己发起的任务。
7. 查看自己发起的历史任务摘要。
8. 查看自己的操作审计日志。
9. 查看小程序名称、任务进度和执行状态。

普通用户不能：

1. 配置数据库。
2. 配置小程序。
3. 设置用户。
4. 导入配置。
5. 导出配置。
6. 查看数据库密码。
7. 查看小程序 AppSecret。
8. 中断、继续、结束别人发起的任务。
9. 查看别人发起的历史任务详情。
10. 查看别人任务的详细日志。

如果别人正在执行任务，普通用户只能看到任务正在执行和整体状态，不能操作该任务。

所有权限必须在后端 API 层强制校验。前端隐藏菜单和按钮只作为用户体验优化。

## 5. 菜单设计

管理员侧边栏菜单：

1. 小程序配置
2. 系统配置
3. 执行拉取
4. 操作日志
5. 用户管理

普通用户侧边栏菜单：

1. 执行拉取
2. 操作日志

### 5.1 小程序配置

仅管理员可见。

功能：

1. 添加小程序。
2. 编辑小程序。
3. 删除小程序。
4. 启用小程序。
5. 禁用小程序。

小程序配置规则：

1. 小程序配置存储在 WMAM 本地系统存储中。
2. `AppSecret` 加密存储。
3. 前端永远不显示完整 `AppSecret`。
4. 编辑小程序时，如果不填写新的 `AppSecret`，保留旧值。
5. 普通用户可在执行拉取页面看到小程序名称和状态，但不能编辑。

### 5.2 系统配置

仅管理员可见。

功能：

1. 数据库连接配置。
2. 测试数据库连接。
3. 恢复上一份可用数据库配置。
4. 导出系统配置。
5. 导入系统配置。

数据库连接配置用于连接广告数据 MySQL 服务器 B。

数据库配置项：

1. MySQL 地址。
2. 端口。
3. 数据库名。
4. 用户名。
5. 密码。

规则：

1. 第一版只支持一套 MySQL 配置。
2. 密码不回显，只显示“已设置”状态。
3. 修改密码时重新输入新密码。
4. 保存数据库配置前必须测试连接成功。
5. 保存前自动记录上一份可用配置。
6. 管理员改错后，可点击“恢复上一份可用配置”。
7. 修改数据库配置需要写入操作审计日志。

### 5.3 执行拉取

管理员和普通用户均可见。

功能：

1. 手动执行拉取。
2. 查看实时任务进度。
3. 查看当前页面内的实时详细日志。
4. 中断任务。
5. 继续任务。
6. 结束任务。

拉取范围：

1. 每次执行默认拉取全部已启用小程序。
2. 普通用户不能选择或排除小程序。
3. 管理员通过小程序启用/禁用控制参与拉取的范围。

普通用户任务权限：

1. 只能操作自己发起的任务。
2. 不能中断别人任务。
3. 不能继续别人任务。
4. 不能结束别人任务。

管理员任务权限：

1. 可以操作任何人的任务。
2. 可以中断、继续、结束任何当前任务。

### 5.4 操作日志

管理员和普通用户均可见，但数据范围不同。

包含：

1. 历史任务摘要。
2. 操作审计日志。

管理员可以查看全部。

普通用户只能查看自己的任务摘要和自己的操作审计日志。

### 5.5 用户管理

仅管理员可见。

功能：

1. 创建普通用户。
2. 启用普通用户。
3. 禁用普通用户。
4. 重置普通用户密码。

不支持：

1. 普通用户公开注册。
2. 创建第二个管理员。
3. 普通用户自行修改角色。

## 6. 初始化流程

首次访问 WMAM 时，如果本地系统存储为空，进入初始化向导。

初始化向导支持两种入口：

1. 创建新系统。
2. 从加密备份文件恢复。

### 6.1 创建新系统

流程：

1. 管理员打开 WMAM。
2. 系统检测到无本地系统存储或未初始化。
3. 显示初始化向导。
4. 管理员填写广告数据 MySQL 连接信息。
5. 点击测试连接。
6. 测试成功后保存数据库连接配置到本地系统存储。
7. 初始化本地系统存储。
8. 创建唯一管理员账号。
9. 生成管理员恢复码。
10. 管理员保存恢复码。
11. 初始化完成，进入登录页。

### 6.2 从备份文件恢复

流程：

1. 管理员选择“从备份文件恢复”。
2. 上传加密备份文件。
3. 输入备份文件密码。
4. 系统解密并校验备份。
5. 校验成功后覆盖本地系统存储。
6. 恢复完成后进入登录页。
7. 使用备份文件中的管理员账号密码登录。

## 7. 登录与密码

### 7.1 登录

登录方式：

1. 用户名。
2. 密码。

登录页支持“记住密码”勾选框。

“记住密码”的用户体验：

1. 勾选后 30 天内无需重复输入账号密码。
2. 文案显示为“记住密码”。

“记住密码”的技术实现：

1. 不保存明文密码。
2. 保存长期登录凭证。
3. 使用 HttpOnly Cookie。
4. 前端 JavaScript 不能读取凭证。
5. 退出登录后凭证失效。
6. 修改密码后所有已登录状态失效。
7. 用户被禁用后所有登录状态失效。

### 7.2 普通用户首次登录

管理员创建普通用户时设置初始密码。

普通用户首次登录后必须修改密码。

流程：

1. 普通用户输入账号和初始密码。
2. 登录成功后只能进入修改密码页面。
3. 修改密码成功后才能进入执行拉取页面。

### 7.3 普通用户忘记密码

普通用户没有自助找回密码入口。

流程：

1. 普通用户联系管理员。
2. 管理员在用户管理中重置密码。
3. 重置后普通用户下次登录必须修改密码。
4. 重置后该用户所有登录状态失效。

### 7.4 管理员忘记密码

系统使用管理员恢复码机制。

规则：

1. 初始化完成后生成管理员恢复码。
2. 管理员需要保存恢复码。
3. 系统只保存恢复码哈希，不保存明文恢复码。
4. 恢复码不能登录系统，只能重置唯一管理员密码。
5. 管理员忘记密码时，可在登录页使用恢复码重置密码。
6. 重置成功后旧恢复码失效。
7. 重置成功后生成新的恢复码。

## 8. 拉取任务设计

### 8.1 任务状态

任务状态：

1. `running`：正在执行。
2. `interrupted`：已中断，可继续。
3. `failed`：执行失败或部分失败，可继续。
4. `ended`：已结束，不可继续。
5. `completed`：全部完成，不可继续。

### 8.2 执行

用户点击“执行拉取”后：

1. 后端检查是否已有任务运行。
2. 如果已有任务运行，返回占用状态。
3. 如果没有任务运行，创建新任务。
4. 获取全局拉取锁。
5. 读取所有已启用小程序。
6. 按小程序和数据类型执行拉取。
7. 实时推送进度和日志到前端。
8. 完成后释放锁。
9. 保存任务摘要。
10. 写入操作审计日志。

### 8.3 中断

中断语义：

1. 停止当前任务。
2. 释放锁。
3. 保留已完成步骤。
4. 任务状态变为 `interrupted`。
5. 允许后续继续。

权限：

1. 普通用户只能中断自己发起的任务。
2. 管理员可以中断任何任务。

### 8.4 继续

继续语义：

1. 只能继续 `interrupted` 或 `failed` 状态的任务。
2. 从失败或未完成步骤继续。
3. 已成功步骤不重复执行。

权限：

1. 普通用户只能继续自己发起的任务。
2. 管理员可以继续任何任务。

### 8.5 结束

结束语义已确认：

1. 终止当前任务。
2. 释放锁。
3. 任务状态标记为 `ended`。
4. 不允许继续。
5. 已成功入库的数据保留。
6. 下次重新点击“执行拉取”时，按数据库已有数据走增量拉取逻辑。

权限：

1. 普通用户只能结束自己发起的任务。
2. 管理员可以结束任何任务。

### 8.6 失败策略

单个步骤失败时：

1. 记录失败步骤。
2. 保存脱敏后的失败摘要。
3. 继续执行后续步骤。

单个小程序失败时：

1. 不阻断其他小程序。
2. 继续拉取后面的小程序。

整个任务完成后：

1. 如果所有步骤成功，状态为 `completed`。
2. 如果存在失败步骤，状态为 `failed`。
3. 界面显示“部分成功/有失败项”。
4. 用户可点击“继续”重试失败或未完成步骤。

## 9. 日志策略

日志分为两类：

1. 执行实时详细日志。
2. 操作审计日志。

### 9.1 执行实时详细日志

执行实时详细日志包括：

1. 正在获取 Token。
2. 正在拉取某个小程序。
3. 某个步骤成功。
4. 某个步骤失败。
5. 微信 API 返回错误。

规则：

1. 通过 SSE 实时推送到前端。
2. 不保存到数据库。
3. 不写入本地系统存储。
4. 页面刷新后消失。
5. 任务结束后，只要当前页面未刷新，详细日志仍然保留在页面上，方便查看。
6. 日志必须脱敏。

脱敏要求：

1. 不显示 `AppSecret`。
2. 不显示 `access_token`。
3. 不显示数据库密码。
4. 不显示包含敏感参数的完整请求 URL。

### 9.2 任务摘要

任务摘要保存到本地系统存储。

保存字段建议包括：

1. 任务 ID。
2. 发起用户 ID。
3. 发起用户名。
4. 任务状态。
5. 开始时间。
6. 结束时间。
7. 耗时。
8. 总小程序数量。
9. 成功步骤数量。
10. 失败步骤数量。
11. 脱敏后的失败原因摘要 `error_summary`。

任务摘要用于历史任务列表。

### 9.3 操作审计日志

操作审计日志永久保存到本地系统存储。

记录事件：

1. 登录。
2. 登出。
3. 创建用户。
4. 启用用户。
5. 禁用用户。
6. 重置用户密码。
7. 添加小程序。
8. 编辑小程序。
9. 删除小程序。
10. 启用小程序。
11. 禁用小程序。
12. 修改数据库配置。
13. 测试数据库连接。
14. 恢复上一份可用数据库配置。
15. 导出系统配置。
16. 导入系统配置。
17. 开始拉取。
18. 中断拉取。
19. 继续拉取。
20. 结束拉取。
21. 任务完成。
22. 任务失败。

管理员可以查看全部审计日志。

普通用户只能查看自己的审计日志。

## 10. 系统配置导入导出

系统配置导入导出只允许唯一管理员操作。

### 10.1 导出系统配置

导出内容：

1. 用户与权限。
2. 用户密码哈希。
3. 小程序配置。
4. 数据库连接配置。
5. 任务摘要。
6. 操作审计日志。
7. 管理员恢复码哈希。
8. 系统状态。

不导出：

1. 明文用户密码。
2. 当前登录状态。
3. 30 天“记住密码”凭证。
4. 执行实时详细日志。
5. 广告数据 MySQL 中的广告明细数据。

导出流程：

1. 管理员点击导出系统配置。
2. 系统要求输入管理员密码确认。
3. 管理员设置备份文件密码。
4. 系统生成加密备份文件。
5. 管理员下载备份文件到本地电脑。

备份文件必须加密，因为其中包含数据库密码、小程序 AppSecret 等敏感信息。

### 10.2 导入系统配置

导入入口：

1. 管理员登录后导入。
2. 系统初始化时从备份文件恢复。

导入流程：

1. 上传备份文件。
2. 输入备份文件密码。
3. 系统解密并校验备份文件。
4. 校验通过后直接覆盖本地系统存储中的配置。
5. 导入后所有登录状态失效。
6. 用户需要重新登录。

已确认导入语义：

1. 直接覆盖即可。
2. 不需要复杂的整库替换流程。
3. 不需要在导入前强制停止并回滚整套系统。

导入后的账号密码：

1. 用户密码哈希会恢复。
2. 用户可继续使用备份时的原密码登录。
3. 如果导入的是旧备份，则管理员需要使用旧备份中的管理员密码登录。

## 11. 数据库连接配置

数据库连接配置用于连接广告数据 MySQL。

已确认：

1. 数据库配置保存在 WMAM 云服务器 A 的本地系统存储。
2. 数据库配置不保存在广告数据 MySQL 服务器 B 中。
3. WMAM 启动后不自动连接 MySQL。
4. 执行拉取时再连接 MySQL。
5. 测试数据库连接时连接 MySQL。
6. 普通用户无法查看或修改数据库配置。
7. 管理员可以在系统配置页面修改数据库配置。
8. 管理员保存配置前必须测试连接成功。
9. 管理员可恢复上一份可用配置。

如果普通用户执行拉取时发现无法连接 MySQL：

1. 实时日志显示连接失败。
2. 普通用户通知管理员。

如果管理员执行拉取时发现无法连接 MySQL：

1. 管理员进入系统配置页面。
2. 修改数据库连接配置。
3. 测试连接。
4. 保存配置。

如果管理员改错数据库配置：

1. 在系统配置页面点击“恢复上一份可用配置”。

## 12. UI 设计方案

### 12.1 整体风格

UI 方向采用：

1. Notion 的结构感。
2. Vercel 的视觉克制感。
3. 少量 Linear 的任务状态交互。

视觉原则：

## 13. 前端技术方案

前端技术栈已确认：

1. React。
2. Vite。
3. Tailwind CSS。
4. shadcn/ui。
5. Lucide Icons。
6. React Router。

选择原因：

1. 用户对 UI 质量要求高。
2. 页面包含主题、折叠侧边栏、初始化向导、Toast、Tab、弹窗、任务状态、实时日志。
3. 继续使用原生 HTML/CSS/JS 后续维护成本会明显升高。
4. React + Vite 可以保持开发效率和可维护性。
5. Tailwind + shadcn/ui 的气质接近 Vercel/Notion/Linear。
6. shadcn/ui 不是黑盒大型组件库，组件代码进入项目后可以按需修改。

不使用：

1. Ant Design。
2. Arco Design。
3. Element Plus。
4. 大而全后台模板。
5. 复杂状态管理库。

路由建议：

```text
/init
/login
/recover-admin
/app/fetch
/app/logs
/app/programs      管理员
/app/system        管理员
/app/users         管理员
```

访问规则：

1. 未初始化时强制进入 `/init`。
2. 未登录时强制进入 `/login`。
3. 已登录普通用户只能访问 `/app/fetch` 和 `/app/logs`。
4. 已登录管理员可访问全部后台页面。
5. 登录后默认进入 `/app/fetch`。
6. 无权限页面显示 403 或重定向到 `/app/fetch`。

构建与托管：

1. 前端由 Vite 构建为静态文件。
2. Go 后端托管构建后的静态文件。
3. 可继续保持单个后端服务对外提供 Web 页面和 API。

## 14. 后端、存储与部署技术方案

### 14.1 后端

后端技术栈：

1. Go。
2. Gin。

选择 Gin 的原因：

1. 当前项目未来可能继续升级。
2. 多角色、多接口、多中间件场景下，Gin 比纯 `net/http` 更易维护。
3. 登录、权限、审计、错误处理、SSE、管理员接口分组都适合用 Gin 管理。
4. Gin 只用于 HTTP 层，业务逻辑不绑定 Gin，方便未来调整。

架构原则：

1. Gin 只负责 HTTP 路由、中间件、请求解析、响应输出。
2. 核心业务逻辑放入独立模块。
3. 拉取逻辑从现有代码中提取复用。
4. 不做微服务。
5. 不引入 Redis。
6. 不做复杂后台任务框架。

第一版不预留 SaaS 租户字段：

1. 不加 `tenant_id`。
2. 不加 `workspace_id`。
3. 先按单组织、唯一管理员、多普通用户实现。
4. 未来真正 SaaS 化时再做明确的数据迁移。

### 14.2 存储

本地系统存储：

1. SQLite。
2. 存储用户、权限、小程序配置、系统配置、任务摘要、审计日志。

广告数据存储：

1. 外部 MySQL。
2. 仅在测试连接或执行拉取时连接。

敏感数据策略：

1. SQLite 文件不整体加密。
2. 用户密码使用 bcrypt 哈希。
3. 管理员恢复码使用哈希。
4. MySQL 密码字段级加密。
5. 小程序 AppSecret 字段级加密。
6. 系统加密密钥保存在云服务器 A 本地。
7. 系统加密密钥包含在加密备份文件中。
8. 备份文件整体使用备份文件密码加密。
9. 备份文件密码无法找回。

### 14.3 部署

第一版部署方式：

1. 不做 Docker 支持。
2. 编译为二进制可执行文件。
3. 上传到云服务器 A。
4. 使用轻量 `config.yaml` 管理运行级配置。
5. 使用 Nginx、Caddy、宝塔、1Panel 等反向代理处理 HTTPS。
6. WMAM 自己只提供 HTTP 服务。
7. 不在 WMAM 内置证书申请和管理。
8. 不做自动更新。
9. 升级通过手动替换程序文件完成。

运行级配置建议：

```yaml
server:
  host: "127.0.0.1"
  port: 28384

data:
  dir: "./data"
```

规则：

1. 没有 `config.yaml` 时使用默认值。
2. 运行级配置不在网页端修改。
3. 网页端只管理系统业务配置。
